# Roadmap PGE5 - Datalyz

## 1. Remaining Features (PGE5 Backlog)
To bring Datalyz to its final PGE5 state, we are evolving the architecture from a standard synchronous API to a distributed, big data processing pipeline. The remaining features include:

* **Real-Time Event Streaming (Apache Kafka):** Implementation of Kafka as a message broker to asynchronously ingest high-throughput data streams from the web extensions.
* **Distributed Data Processing (Spark & Python Workers):** Deployment of Python worker instances utilizing Apache Spark to consume Kafka messages, performing heavy data transformations and advanced analytics.
* **High-Volume NoSQL Storage (Apache Cassandra):** Integration of Cassandra to handle the massive write-throughput of time-series/event data, complementing our existing PostgreSQL metadata database.
* **Analytics Dashboard (Frontend):** Finalizing the user interface (Vue.js/Vite) to visually display the refined insights generated by the Spark workers.

## 2. Technical Challenges
The integration of this advanced data stack introduces significant technical complexity:

* **Distributed Architecture Orchestration:** Seamlessly connecting the Go API gateway, Kafka brokers, Python/Spark workers, and multiple databases within our Dockerized CI/CD pipeline.
* **Data Consistency & Pipeline Latency:** Guaranteeing message delivery from the Go API to Kafka, and ensuring the Python workers process and write to Cassandra without bottlenecks.
* **Resource Management:** Managing the high memory and CPU overhead of running JVM-based infrastructure alongside our lightweight Go backend on our deployment servers.

## 3. Risks and Mitigation Strategies
We have identified the following risks for the PGE5 phase and established clear mitigation plans:

| Risk Area | Identified Risk | Mitigation Strategy |
| :--- | :--- | :--- |
| **Infrastructure Overhead** | Resource exhaustion on the Debian VM causing crashes due to heavy Kafka/Cassandra/Spark instances. | Implement strict Docker resource limits. Prepare a scaling plan to migrate to a multi-node infrastructure if necessary. |
| **System Complexity** | The addition of new heavy components delays the release due to integration bugs. | Follow the strict 7-step action plan. Validate data ingestion before refining the algorithm. |
| **Data Privacy** | Inadvertently exposing user-sensitive data as it flows through the distributed pipeline. | Enforce strict network isolation (internal Docker networks) and anonymize data at the Go API level before publishing to Kafka. |

## 4. Planning and Prioritization (Action Plan)
Our PGE5 development cycle follows a strict 7-step timeline from July to January to safely roll out the distributed architecture.

* **1 --> Event: Oral Presentation (July)**
    * Presentation of the current architecture and validation of the Greenlight. Kick-off for the new Big Data stack architecture planning.
* **2 --> Event: Sampling / Proof of Concept (August)**
    * Initial deployment of the new stack (Kafka, Cassandra). Testing the Go backend as a Kafka producer with small sample data sets to ensure stable connectivity.
* **3 --> Event: Marketing & Communication Strategy (September)**
    * Definition of the public technical valorization strategy and preparation of communication materials regarding our robust data pipeline.
* **4 --> Event: Data Ingestion (October)**
    * Full-scale deployment of the web extension data collection feeding directly into Kafka. Verification of high-throughput data writing into Cassandra.
* **5 --> Event: Algorithm Refinement (November)**
    * Deployment and tuning of the Python/Spark workers. Processing the raw ingested data from Kafka, applying our analytical algorithms, and structuring the refined outputs for the frontend.
* **6 --> Event: Security Audit (December)**
    * Comprehensive security review of the entire distributed architecture. Checking firewall rules, internal network isolation, and data anonymization across all instances.
* **7 --> Event: Finalization (January)**
    * Connection of the Vue.js dashboard to the refined data endpoints. Final UI polish, bug fixing, and codebase freeze for the final PGE5 delivery.
